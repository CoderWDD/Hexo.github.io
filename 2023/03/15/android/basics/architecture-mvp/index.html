MVP 简介

MVP 三个字母分别代表什么

  • M：Model，包含了一些处理逻辑和数据定义
  • V：View，用于处理用户输入和显示，负责用户交互
  • P：Presenter，连接 Model 和 View 层，接收来自 View 的事件，并分发给不同的 Model，最后调用 View 层的方法，更新 View，与 Controller 的区别在于 Presenter 完全独立出来了

MVP 三层之间的联系

  • Model 层负责定义数据类型，接收来自 Presenter 的事件， 并进行相关的逻辑处理，并在处理完数据后，把结果返回给 Presenter 层
  • View 层负责与用户交互，并把相关的事件传递给 Controller 层
  • Controller 层负责接收来自 View 层的事件，并调用 Model 层的处理，把结果组合后返回给 View 层使用

注意点

  • 臃肿、难维护：从上面的例子中可以看出 Presenter 层，其实就是独立出来的 Controller，所以 Controller 的缺点对于 Presenter 仍然存在，这意味着 Presenter 仍然具有 臃肿、难维护 的缺点
  • 责任不够单一：在 Presenter 中，我们只希望 Presenter 的责任是负责桥接 View 和 Model 层，但实际在每次需要更新 View 时，Presenter 还得承担起调用 View 层的方法来更新 View 的责任，这使得 Presenter 的责任并不单一

总结

MVP 模式在 MVC 的基础上，解决了 View 层与 Controller 层的耦合，避免了 Activity 过于臃肿的问题，但是，也只是把相应的任务都抽离给了 Presenter 层而已，对于 Presenter 层来说，任务还是很重，而且因为持有了 View 层的引用，使得 Presenter 的 责任并不单一。而且对于 MVP 模式来说，因为独立了 Presenter 出来，也导致了其使用起来比 MVC 更加麻烦
