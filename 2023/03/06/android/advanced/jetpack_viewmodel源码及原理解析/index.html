简介

The ViewModel class is a business logic or screen level state holder. It exposes state to the UI and encapsulates related business logic. Its principal advantage is that it caches state and persists it through configuration changes.

  • 首先，ViewModel 是一个状态持有者，只要把状态保存在 ViewModel 中，那状态数据的存活周期，就与 App 应用是一个级别的。所以官网说是一个 screen level state holder
  • 同时，ViewModel 也是我们在讨论 MVVM、MVI 这类应用架构时，其中的 VM 层，是连接 视图 View 和 Model 的中间层，其实就是我们写业务代码，处理业务逻辑的地方，所以，这也是一个 business logic holder

原理

  • ViewModel 通过把 数据/状态，保存在 ViewModelStore 中，而 ViewModelStore 则是保存在 Activity 的 静态 内部类 NonConfigurationInstances 中，因为 静态类 的生命周期的特点，所以，ViewModel 的生命周期可以横跨整个 APP
  • 在页面发生类似切换横竖屏时，虽然 Activity 会被 destroy，但是 ViewModel 不会，可以从 NonConfigurationInstances 的 ViewModelStore 中取出，从而获取旧数据
  • 而 NonConfigurationInstances 是每个 Activity 的静态内部类（具体看代码），这意味着，每个 Activity 都单独对应着一个 ViewModelStore，即 ViewModel 设计的初衷就不打算让多个 Activity 直接共享

总结

  • ViewModel 其实并没有用很神奇高深的技术，本质上就是以键值对的形式，把 ViewModel 实例存储起来，放到 ViewModelStore 中，而 ViewModelStore 又会放在每一个 Activity 的静态内部类 NonConfigurationInstances 中，而在 Activity 重建时，会从旧的 Activity 传回一个 NonConfigurationInstances 实例，其中保留着旧 Activity 中的 ViewModelStore，从而把 ViewModel 继承过来
  • 同时，因为 ViewModel 没有实现 lifecycle 的相关方法，所以不能做到自己管理生命周期，需要配合在 Activity 的生命周期方法中进行相应的调用，才能保证不会发生意外，好在官方已经把这部分工作做了，我们只需要自己实现其 onClear() 方法，进行释放一些在 ViewModel 引用的资源即可

问题解答

  • ViewModel 生命周期为什么可以横跨整个 Application
    • 因为 ViewModel 是存储在 ViewModelStore 中，而 ViewModelStore 是存在 Activity 的静态内部类 NonConfigurationInstances 中的，而 NonConfigurationInstances 实例会在旧 Activity 中传给新的 Activity，也就是说，ViewModel 可以跨越 Activity 的生命周期，达到横跨整个 Application 的生命周期
  • ViewModel 是怎么实现在一个 Activity 被销毁后，仍然能在新的 Activity 中保存状态的
    • 旧的 Activity 会把持有 ViewModelStore 的 NonConfigurationInstances 实例传给新的 Activity
  • ViewModel 对象，在 Activity 重建之后，什么时候被恢复
    • 在新的 Activity 的 onRetainNonConfigurationInstance() 被调用时恢复
  • 什么情况下，ViewModel 也会被清除
    • 被动：因为 ViewModel 的生命周期横跨整个 Application，所以，只有在 APP 被 销毁/异常退出 时，才会被清除
    • 主动：调用 ViewModel 的 clear() 方法
  • ViewModel 什么情况下会内存泄漏
    • 持有其他需要被回收的对象的引用，将会导致其无法被回收，最终内存泄漏
  • ViewModel 如何避免内存泄漏
    • 避免持有非必须的对象
    • 在不需要用到时，调用 clear() 方法，并重载 onClear() 方法，在其中释放相应的资源
  • 如果ViewModel的构造器需要传入参数，需要怎么做
    • 自定义 ViewModelProvider.Factory，并在其 create() 方法中，返回对应的 ViewModel 实例
  • ViewModel 有几种构建的方法
    • 一种，使用 ViewModelProvider(ViewModelStoreOwner).get(ViewModel.class) 创建
  • 如果不自定义，同一个 ViewModel 实例，可以在多个 Activity 之间共享吗？
    • 不行，因为最终持有 ViewModelStore 的 NonConfigurationInstances 是 ComponentActivity 的静态内部类，是每个 Activity 实例独有的，没法共享
  • 同一个 ViewModel 实例，可以在多个 Fragment 之间共享吗？
    • 可以，在构建 ViewModel 实例时，把传入 ViewModelStoreOwner 指明为多个 Fragment 所属的 Activity 即可
  • 如果一定要实现 ViewModel 在多个 Activity 直接共享，应该怎么办？
    • 因为 ViewModelStore 类是 public 的，所以可以自己继承 ViewModelStore 类，并把对应的实例声明为 Application 级别的，就可以实现在多个 Activity 乃至整个 App 中共享了。
